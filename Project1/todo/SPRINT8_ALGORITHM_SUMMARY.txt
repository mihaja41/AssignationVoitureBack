================================================================================

MODIFICATION 1 - REGROUPEMENT OPTIMAL
---------------------------------------
Règle: Regrouper les réservations dans LE MÊME VÉHICULE en trouvant celle
       avec un nombre de passagers CLOSEST to remaining vehicle capacity.

Concept: "écart minimum" = Math.abs(nb_passagers_restants - places_restantes)
         Minimiser cet écart = meilleure utilisation du véhicule

Implémentation:
- Méthode: trouverMeilleureReservationPourRegroupementOptimal() (ligne ~1648)
- Boucle sur toutes les réservations non assignées
- Calcul de l'écart pour chaque
- Sélection de celle avec écart minimum
- En cas d'égalité (tie-breaking):
  * Moins de trajets (DB + session)
  * Carburant Diesel
  * Aléatoire


MODIFICATION 2 - DIVISION AVEC SÉLECTION VÉHICULE OPTIMALE
-----------------------------------------------------------
Règle: Quand une réservation ne peut pas tenir ENTIÈRE dans UN seul véhicule,
       DIVISER les passagers entre PLUSIEURS véhicules.
       Choisir le véhicule avec nb_places CLOSEST to remaining passengers
       (encore une fois, l"écart minimum).

Implémentation:
- Méthode: trouverMeilleureAttributionAvecDivision() (ligne ~684)
- Méthode helper: selectionnerVehiculeOptimalPourAssignation()
- BOUCLE de division:
  1. Chercher le véhicule optimal pour les passagers restants
  2. Assigner ce qui RENTRE dans ce véhicule
  3. Mettre à jour passagersDejaAssignes
  4. Créer une Attribution pour cette assignation
  5. Répéter jusqu'à tous les passagers assignés

Résultat: UNE réservation peut être divisée en PLUSIEURS véhicules


MODIFICATION 3 - DISPONIBILITÉ VÉHICULE PAR HEURE
---------------------------------------------------
Règle: Véhicule disponible seulement après son heure_disponible_debut
       (nouveau champ depuis  sprint 7).

Implémentation:
- Vérification: heure_retour >= vehicule.heure_disponible_debut
- Appliquée APRÈS faire l'assignation, avant confirmer dans l'attribution finale

2. LES QUATRE BUGS CRITIQUES CORRIGÉS
================================================================================

BUG 1 - DOUBLE COMPTAGE DE PASSAGERS APRÈS REGROUPEMENT
-------------------------------------------------
Symptôme: Première attribution affichait 20 passagers au lieu de 12
Root cause: Attribution utilisait getReservations().stream().sum() pour
            compter les passagers même quand une réservation était
            PARTIELLEMENT assignée par regroupement.

Correction:
- Ajouté Map<Long, Integer> passagersParReservation dans Attribution
- Méthodes: getPassagersPourReservation(), setPassagersPourReservation()
- À chaque assignation partielle, stocker le nombre EXACT de passagers
  dans CE véhicule pour CETTE réservation
- result.jsp utilise maintenant: attr.getPassagersPourReservation(r.getId())
  au lieu de simplement r.getPassengerNbr()


BUG 2 - SÉLECTION INCORRECTE DE VÉHICULE À 13:30
--------------------------------------------------
Symptôme: vehicule3 choisi pour client6 à 13:30 au lieu de vehicule5
Root cause:
- vehicule3 revenant à 13:00/13:06 au lieu de vehicule5 à 13:30
- Pas de tri-breaker pour différencier les véhicules équivalents
- Pas de suivi des trajets effectués pendant la SESSION courante

Correction:
- Ajouté Map<Long, Integer> trajetsSessionParVehicule
- Incrémentée pour chaque véhicule à chaque fenêtre (ligne 145)
- Combinée avec trajets en BD dans le tri-breaker:
  totalTrajets = trajetsDB + trajetsSession
- Véhicule avec MOINS de trajets totaux = prioritaire
- Diesel après le nombre de trajets
- Aléatoire comme dernier tri-breaker


BUG 3 - CLIENTS RESTANTS NON ASSIGNÉS À 13:30
----------------------------------------------
Symptôme: client5 restant (2 passagers) pas assigné à vehicule1 à 13:30
Root cause:
- Après regroupement sans division, si passagers restent non assignés
  et aucun véhicule disponible pour eux, on marquait la réservation
  comme "complètement non assignée" au lieu de "partiellement assignée"

Correction:
- Créer une ReservationPartielle même en regroupement (lignes 325-329)
- Cette partielle est reportée automatiquement à la fenêtre suivante
- À la fenêtre suivante (13:30), elle est traitée normalement
- Les 2 passagers restants trouvent vehicule1 disponible


BUG 4 - VÉHICULES REVENANT TROP TÔT INCLUS COMME DISPONIBLES
-------------------------------------------------------------
Symptôme: vehicule3 revenant à 13:00 considéré comme disponible à 09:24
Root cause:
- Condition originale trop permissive: !heureRetour.isBefore(startTime)
- Acceptait les véhicules revenant à 13:00, bien hors de fenêtre [09:24, 13:00]

Correction:
REGROUPEMENT (ligne ~586-591):
  Au lieu de: !heureRetour.isBefore(startTime)
  Changer à: !heureRetour.isAfter(endTime)
  = Véhicule DOIT revenir PENDANT OU AVANT la fin de fenêtre

DIVISION (ligne ~742-748):
  Au lieu de: !heureRetour.isBefore(startTime)
  Changer à: !heureRetour.isAfter(dateHeureDepart)
  = Véhicule DOIT revenir AVANT la prochaine assignation

7. LESSONS LEARNED
================================================================================

1. TRACKING DISTRIBUÉ EST CRITIQUE
   → Map<Long, Integer> passagersDejaAssignes à fenêtre level
   → Map<Long, Integer> passagersParReservation à attribution level
   → Chacune représente "la vérité" pour son contexte

2. ÉCART MINIMUM EST L'OPTIMISATION CLÉE
   → Math.abs(nb_restants - places_restantes) guide TOUTE la sélection
   → Applicable à regroupement ET division

3. TIE-BREAKING HIERACHIQUE IMPORTANT
   → Écart → Trajets → Diesel → Aléatoire
   → Chaque niveau est décisif quand les niveaux précédents sont égaux

4. PARTIAL RESERVATIONS BRIDGE LES FENÊTRES
   → Une réservation peut être divisée ENTRE fenêtres
   → Créer ReservationPartielle = contrat pour fenêtre suivante
   → Auto-reportage via creerReservationPourFenetresuivante()

5. FENÊTRE D'HEURE EST UN HARD CONSTRAINT
   → Mais doit être définie précisément pour regroupement vs division
   → Regroupement: véhicule revient DANS la fenêtre
   → Division: véhicule revient AVANT le départ suivant

6. SESSION-LEVEL STATE MATTERS
   → trajetsSessionParVehicule essentiels pour tie-breaking
   → DB trajets insuffisant (pas de context de session courante)
